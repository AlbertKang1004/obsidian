## User Story
>**User stories** capture user needs in simple, concise statements focusing on the desired feature and benefit.

**Example** (To-do calendar app)
- **As a user**,  I want to add a task with a title and due date
- **So that** I can keep track of what I need to do

**Example** (Social app)
- **As a user**, I want to send friend requests to other users
- **So that** I can connect and share content with them

## Use Case
### Key Elements of a Use Case
1. **Use Case Title**
	- The title clearly defines the primary goal or purpose of the use case for easy identification.
2. **Primary Actor and Preconditions**
	- The primary actor initiates the use case
	- Preconditions specify required states before execution.
3. **Main Flow and Alternative Flows**
	- Main flow details standard steps.
	- Alternative flows represent variations in the process.
4. **Exception and Postconditions**
	- Exception flows handle errors.
	- Postconditions define system state after use case completion.
### Example
>**User Story** : As a registered user, I want to reset my password so that I can regain access to my account if I forget it.

- **Use Case Title**: Reset Password
- **Primary Actor**: Registered User 
- **Preconditions**: 
	1. User has an existing account.
	2. User has access to the email associated with the account.

**Main Flow**
- User clicks on "Forgot Password" link.
- System prompts for the registered email address.
- User enters their email.
- System sends a password reset link to the email.
- User clicks the link and is taken to a password reset page.
- User enters a new password and confirms it.
- System updates the password and confirms success.

**Alternative Flows**
- Email is not registered → System displays an error message.
- Passwords do not match → System prompts user to re-enter.

**Postconditions** 
- User's password is updated.
- User can log in with the new password.
## User Story vs. Use Case

| Feature      |  User Story   |                Use Case                 |
| ------------ | :-----------: | :-------------------------------------: |
| Format       | One sentence  |            Structured steps             |
| Detail Level |  High-level   |                Detailed                 |
| Purpose      | Capture needs | Guide design, <br>programming & testing |
| Audience     | Stakeholders  |           Developers, Testers           |
## Stakeholder
>**Stakeholder**s are anyone who has an interest in or is affected by the program.

| Stakeholder            | Layer in CA            | Example Role in System                                                   |
| ---------------------- | ---------------------- | ------------------------------------------------------------------------ |
| Customer               | View                   | Places orders, views products                                            |
| Admin / Manager        | Interface / Controller | Manages inventory, monitors sales                                        |
| Payment Gateway        | Interface Adapter      | Processes payments via API                                               |
| Order Processing Logic | Use Case Interactor    | Executes core business rules like order validation, discount calculation |
| Product Entity         | Entity                 | Represents products, pricing, stock                                      |
| IT / DevOps            | Frameworks             | Maintains database, deployment, logging                                  |
